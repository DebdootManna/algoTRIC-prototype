Abstract

algoTRIC studies the trade-offs between symmetric and asymmetric cryptography in modern AI-driven systems. This page summarizes the research, explains the demo prototype (an AES + RSA hybrid pipeline), and connects the implementation to matrix-and-transform schemes described in the companion paper.

1. What the prototype implements

The repository included a compact, well-documented demo that implements a classic hybrid encryption pattern:

  1. Generate an ephemeral AES key and encrypt the bulk message using AES-GCM.
  2. Encrypt the AES key using the recipient's RSA public key (RSA-OAEP).
  3. Bundle the RSA-encrypted AES key with AES nonce, ciphertext and tag into a payload.
  4. On the receiver side, RSA-decrypt the AES key and use it to decrypt the ciphertext.

Why hybrid?

Hybrid encryption leverages the performance of symmetric ciphers for large payloads and the secure key distribution of asymmetric ciphers for session key exchange. The prototype benchmarks the extra cost added by the RSA step and demonstrates the trade-offs discussed in the paper.

Representative code snippet

// simplified pseudo-code from the demo
// AES encrypt
key, nonce, ct, tag = aes_encrypt(plaintext)
// RSA wrap the AES key
enc_key = rsa_encrypt(pubkey, key)
// payload = enc_key || nonce || ct || tag

// decryption
key = rsa_decrypt(privkey, enc_key)
plaintext = aes_decrypt(key, nonce, ct, tag)

2. How this maps to the algoTRIC papers

Paper A (algoTRIC — SE vs AE): the repository is a direct, practical instantiation of the SE vs AE discussion. The benchmark script compares AES-only to AES+RSA hybrid performance across message sizes — this concretely demonstrates the paper’s main trade-off: speed vs key management/security.

Paper B (Matrix & Transformation-based cryptography): not yet implemented in the prototype. The paper proposes converting text into matrices, using LU decomposition, and applying transforms (Gupta and Abaoub-Shkheam) as part of encryption/decryption. Extending the repo to include a matrix-transform cipher would make the implementation fully reflect both papers.

3. Reproducing the demo (quick)

Minimum steps to run locally (from repository root):

python -m venv .venv
source .venv/bin/activate
pip install -r requirements.txt
# run an interactive demo
python -m src.crypto.aes_rsa_hybrid --demo
# run benchmarks
to run: python bench/benchmark.py --sizes 1024 10240 --iters 10
# plot results
python bench/plot_results.py bench/results.json

Tip: use Python 3.10+ and a machine with enough RAM to measure the larger sizes reliably.

4. Benchmarks — what to expect

The included benchmark compares AES-only to the hybrid approach across sizes. Typical expectations:

  • For small messages (< few KB) RSA wrapping dominates overhead — hybrid is significantly slower than AES-only.
  • For large messages (MBs), AES encryption time dominates, and the relative overhead of RSA wrapping becomes negligible.
  • GCM adds integrity (tag verification) and has minor CPU overhead compared with basic CBC + HMAC patterns.

Suggested plots: time vs message size (log scale), CPU/RAM usage during encrypt/decrypt. The repo already saves JSON and PNGs.

5. Roadmap & suggested extensions

  1. Matrix-transform cipher: implement the LU + Gupta/Abaoub-Shkheam pipeline. Add it as a symmetric layer alternative and benchmark against AES.
  2. ECC hybrid: add ECDH-based key encapsulation (faster and smaller keys than RSA).
  3. AI-driven tuning: simple ML model to predict optimal key sizes/parameters based on message size & device profile.
  4. Secure packaging: adopt an authenticated file format (header + JSON metadata + binary sections) and authenticated encryption for metadata.
  5. CI & reproducible experiments: GitHub Actions to run tests and a small benchmark on every push; use Docker for exact environment reproduction.

6. Short glossary

AES-GCM — Advanced Encryption Standard in Galois/Counter Mode: provides confidentiality + integrity.

RSA-OAEP — RSA encryption with Optimal Asymmetric Encryption Padding: secure for key encapsulation.

LU decomposition — factorization of a matrix into Lower and Upper triangular matrices; used in the matrix-transform proposal.

Gupta / Abaoub-Shkheam transforms — integral transforms used in the matrix-transform paper to perform further obfuscation / transform-domain encryption.
